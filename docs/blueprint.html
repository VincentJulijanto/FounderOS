1. Executive Overview

FounderOS is an AI board for people who already run a business. The highest-stakes decisions (enter a new market, hire a senior leader, take on debt, sunset a product line) usually get made with whoever happens to be in the room. Co-founders share your blind spots. Direct reports won't tell you an idea is bad. A real board meets quarterly and lacks the context. A consultant costs six figures and takes weeks to hand back a slide deck instead of a decision. So the call gets made on instinct and the loudest voice, and the risk no one considered only surfaces once the capital is spent. FounderOS gives that operator a standing board of seven specialist agents. It takes one decision ('is this sound, and what are we missing?'), decomposes it, argues it from every angle, and returns a board-ready memo: a clear recommendation, the dissent behind it, what's still unknown, and a phased plan. On demand, in minutes, at software cost.

Why a board, not a chatbot

Ask a single AI 'should we expand into Indonesia?' and it gives a fluent, agreeable answer: it pattern-matches toward yes and never argues with itself. That is the yes-man failure mode, automated, and it is the same reason asking one advisor or a room that defers to the CEO is dangerous: nothing pressure-tests the decision. A society of agents with opposing mandates does. Finance argues the economics, Growth argues the upside, the Skeptic's only job is to attack, and Capability asks whether the organization can actually execute. They disagree, and the disagreement is the product. The Debate Engine forces each position to survive rebuttal across multiple rounds, so what reaches you is not one model's guess but a recommendation that has already withstood its own strongest counterarguments, with the minority view recorded so you see what the board was divided on, not just the verdict. And because the board remembers (every prior decision and its outcome is written back to per-company memory), its judgment compounds over time, the way a long-serving board's would, instead of resetting with every conversation.

The transformation we deliver:

From
"Should we expand into Indonesia next quarter, or deepen in our home market first?"
To
"A board memo: a clear recommendation (proceed / hold / conditional), the dissent behind it, what's still missing, and a phased plan, pressure-tested by 7 specialist agents."

The Vault: the board's institutional memory

A real board is valuable because it remembers. It knows what the company has already tried, how it tends to decide, and how the last big call turned out. FounderOS gives the agents that same memory through a per-company vault of markdown notes that every run reads from and writes back to, so the board reasons with accumulating context instead of starting cold each session.

What the vault holds is chosen for what actually changes a recommendation, not everything about the company:

  • Company profile: sector, stage, business model, and size, so every agent argues about the real business rather than a generic one.
  • Financial shape: revenue band, margin, and cash position, the constraints Finance reasons against.
  • How it operates: risk appetite, decision-making style, and what the company has said no to before, which is what the Capability agent reads to judge whether the organization can actually execute.
  • Decision history: every past decision, the recommendation given, the recorded dissent, and, once known, the outcome. This is the highest-value part. It lets the board learn that a similar expansion was approved before and underperformed, and weight the next call accordingly.
  • Durable market and context notes gathered along the way.

What it deliberately leaves out is narrative that does not move a decision: founder biographies, mission-statement prose, and anything the analysts never consume. The same discipline the intake form uses (keep what an agent reasons over, drop the rest) keeps the memory sharp and the context small enough to stay fast.

4. The Agent Society: Roles & Task Division

Seven agents, each a single responsibility, now company-centric (evaluating one decision, not inventing ideas). This is the task decomposition Track 3 rewards.

AgentThe question it owns, and what it protects you fromStructured Output
Opportunity Scout (scout)"What are the real options here?" Frames the alternatives (and generates them only if none are given). Protects against tunnel-vision on the first idea that happened to be in the room.Options assessed
Trend Analyst (trend)"Is the market and the timing actually there?" Reads demand with live MCP signals. Protects against committing to a decision whose demand was only assumed.Market read
Finance (finance)"What does this do to the P&L, margin, and cash?" Protects against a move that looks strategic but the balance sheet cannot survive.Plain-language financial view
Growth (growth)"If we commit, how do we actually execute and win customers?" Protects against a yes with no credible route to results.Execution / GTM read
Skeptic (skeptic)The main event. "What is the strongest case that this fails?" Attacks the assumptions and failure modes. Protects against the risk no one in the room wanted to raise.Risk report
Capability (capability)"Can this organization actually pull it off?" Weighs team, ops, and track record (not a person's skills). Protects against a sound idea the company is not staffed or ready to run.Capability read
Chair (venture_partner)"Given all of the above, what do we do?" Reconciles the debate and weights the Skeptic explicitly. Protects against a call decided by the loudest voice instead of the balance of argument.Board memo (BoardRecommendation)

Canonical agent-name strings (roster ↔ agent_name ↔ graph node key ↔ summary key): scout · trend · finance · growth · skeptic · capability · venture_partner. capability is the rebuilt founder_fit (org readiness, not a person). The synthesizer displays as "Chair"; its canonical string stays venture_partner.

Conflict Resolution Protocol (the core innovation)

  1. Independent analysis: agents produce positions without seeing each other.
  2. Conflict detection: the engine flags contradictions (e.g. Trend says "enter now" vs Finance "runway can't support two markets").
  3. Debate rounds: conflicting agents exchange rebuttals; each may revise its stance (up to N rounds).
  4. Consensus scoring: quantify agreement; unresolved tensions are surfaced as the dissent record, not hidden.
  5. Final reconciliation: the Chair (venture_partner) issues the recommendation, weighting the Skeptic explicitly.

The entire system is built to run end-to-end before any real API key exists. Keys are injected purely through environment variables, and every external call has a deterministic mock fallback.

5.2 Provider layer with mock fallback: backend/llm/provider.py

from openai import OpenAI
from ..config import settings

class QwenProvider:
    def __init__(self):
        self.mock = not settings.qwen_api_key      # no key → mock mode
        if not self.mock:
            self.client = OpenAI(
                api_key=settings.qwen_api_key,
                base_url=settings.qwen_base_url,   # OpenAI-compatible endpoint
            )

    def chat(self, system: str, user: str, max_tokens: int = 2000) -> str:
        if self.mock:
            return _mock_response(system, user)    # deterministic fixture
        resp = self.client.chat.completions.create(
            model=settings.qwen_model,
            response_format={"type": "json_object"},  # structured output
            messages=[{"role":"system","content":system},
                      {"role":"user","content":user}],
            max_tokens=max_tokens,
        )
        return resp.choices[0].message.content

Result: uvicorn + npm run dev work immediately with mock data. The day a QwenCloud key lands, you paste it into .env, set USE_MOCK_LLM=false, and the same code path goes live, with zero refactor.

6. The Build, Phase by Phase (Start → Finish)

The order matters: the system runs end-to-end in mock mode from Phase 1, and each phase adds one capability without breaking the last. Every phase names a concrete deliverable you can demo.

Live build status and the task split between the two builders live in PROJECT_STATE.md (Build Tracker).

Phase 0Project Scaffold & Tooling

  • Repo layout: backend/ (FastAPI), frontend/ (Next.js), docs/.
  • requirements.txt: fastapi, uvicorn, pydantic, openai (Qwen-compatible), langgraph, httpx, pytest. (No sqlalchemy/asyncpg: the Postgres modules are unwired; memory is the vault.)
  • Commit .env.example, add .env to .gitignore.
  • Deliverable: uvicorn backend.main:app returns a health check; npm run dev renders a shell.

Phase 1Provider & Mock Layer scaffolding core

  • Build QwenProvider with mock fallback (§5.2) and JSON-mode parsing utilities.
  • Author deterministic mock fixtures for every agent so the full pipeline runs key-free.
  • Deliverable: end-to-end /api/analyze returns a complete (mocked) BoardResponse.

Phase 2Agent Abstraction & the Seven Agents

  • BaseAgent: shared provider, profile formatter, JSON parsing, retry-on-malformed-JSON.
  • Implement Scout, Trend, Finance, Growth, Skeptic, Capability, and the Chair (venture_partner), each a focused system prompt + typed output.
  • Deliverable: each agent independently produces a validated AgentOutput.

Phase 3LangGraph Orchestration (task division)

  • Define a typed graph state; nodes = agents, edges = data flow.
  • Fan-out the four analysts in parallel (performance optimization), then converge.
  • Deliverable: orchestrator runs Scout → parallel analysts → Skeptic deterministically.

Phase 4Debate & Consensus Engine (negotiation)

  • Conflict detector → N debate rounds → consensus scorer (the custom algorithm judges reward).
  • Persist every round so the UI can replay the negotiation.
  • Deliverable: measurable consensus score; unresolved conflicts surfaced.

Phase 5Memory Loop: per-company vault

  • Wire the per-company Obsidian markdown vault: read(company_id, query) does LLM-driven selective note-retrieval over a small index (frontmatter + filename + one-line summary), with no embeddings and no vector DB, then feeds the relevant context in.
  • write_back records the decision + recommendation, and later its outcome, closing the validation loop.
  • Deliverable: a returning company's later decisions draw on its prior decisions and outcomes.

Phase 6MCP Integration (live signals)

  • Connect an MCP server so Trend Analyst pulls real market/web signals via tool calls.
  • Mock the tool responses until the MCP endpoint key is supplied.
  • Deliverable: Trend analysis cites live data when MCP is configured.

Phase 7Frontend: Visualize the Society

  • Four-phase UI: Profile → Analyzing → Debating → Results.
  • Agent cards, live conflict badges, consensus meter, execution-plan renderer.
  • Deliverable: the debate is visible, which directly scores the Presentation pillar.

Phase 8Benchmark Harness (efficiency gain)

  • Single-agent baseline vs full society on the same profiles; log the metrics in §8.
  • Deliverable: a chart proving the society beats one agent, Track 3's explicit ask.

Phase 9Go Live: Insert Keys

  • Paste QWEN_API_KEY + MCP_SERVER_URL into .env; set USE_MOCK_LLM=false.
  • Deploy per Decision #8: backend to Hugging Face Spaces (Docker, port 7860, seed vault baked in), frontend to Vercel Hobby; set VAULT_PATH, ALLOWED_ORIGINS, and the frontend's NEXT_PUBLIC_API_BASE_URL (see docs/deployment.md).
  • Run benchmark again with the real model; record final numbers for the demo.
  • Deliverable: production pipeline on real QwenCloud inference.

7. QwenCloud & MCP: Technical Depth Details

Two places carry the technical weight: how the agents use the model, and how they reach live market data. Both degrade gracefully: they run on deterministic mocks until real keys are supplied.

7.1 Sophisticated QwenCloud usage

  • OpenAI-compatible SDK against the DashScope endpoint: one client, seven agents.
  • JSON mode (response_format=json_object) for reliable structured outputs.
  • Tool / function calling in the Trend Analyst to invoke MCP market lookups.
  • Model tiering: qwen-plus for reasoning-heavy agents, qwen-turbo for cheap fan-out, a cost/perf optimization.
  • Response caching keyed on (agent, profile-hash) to cut latency & spend on repeat runs.

7.2 MCP integration sketch

10. Quickstart (runs key-free in mock mode)

The whole system runs without a single API key (mock fixtures stand in for every external call), so a reviewer can clone, run, and see a complete board memo in minutes.

# 1. Backend
cd backend && python -m venv venv && source venv/bin/activate
pip install -r requirements.txt
cp ../.env.example ../.env        # leave keys blank for now
uvicorn backend.main:app --reload --port 8000

# 2. Frontend
cd frontend && npm install && npm run dev    # http://localhost:3000

# 3. When QwenCloud key is ready:
#    edit .env → QWEN_API_KEY=... and USE_MOCK_LLM=false → restart. Done.
